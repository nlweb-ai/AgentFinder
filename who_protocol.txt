Who Protocol Specification
Version: 0.1 (Draft)
 Date: January 2026
 Status: Proposal

Abstract
Who is a discovery service for capability augments—MCP tools, A2A agents, skills, and other callable services that extend the capabilities of large language models. Given a natural language query, Who returns a ranked list of augments suited to handle the request, along with sufficient information to invoke each one.
Who can be invoked as a service, enabling any LLM-powered system to dynamically discover and utilize relevant capabilities without requiring users to pre-install them.

1. Motivation
LLMs increasingly rely on external "capability augments"—MCP tools, skills, or the ability to invoke other agents—to extend their functionality. In this document, we refer to these generically as augments.
Today's approach to augments faces fundamental challenges in discovery and selection: how does an LLM find relevant augments, and how does it choose among them?
1.1 The Selection Problem
The prevailing model requires users to explicitly "install" each augment before use. For each request, the LLM then selects from installed augments and composes invocations based on their schema descriptions. This approach encounters serious scaling limitations:
The installation model is ill-suited to the task. We will see thousands—likely millions—of augments, especially agents. Requiring users to install agents before use replicates the mobile app store paradigm rather than the open web. It is analogous to requiring users to bookmark a URL before they can visit it, rather than discovering it through search. On the web, search enables users to find relevant sites for one-time needs far more effectively than curated bookmark lists. We need a similar model for augments: one where LLMs can draw from an open repository without requiring pre-installation.


Augment descriptions lack sufficient detail for accurate selection. Agent descriptions tend to be generic and omit the information needed for precise matching—a problem that intensifies as augments compete for usage. Effective selection requires richer signals: representative queries the augment handles well (or poorly), domain expertise indicators, and success patterns. However, embedding this information in prompts only exacerbates the context window problem.


Context-based selection does not scale. Most LLMs select tools by including tool descriptions in the context window, which incurs quadratic computational cost and cannot accommodate thousands of tools. Recent work on hierarchical tool directories offers modest improvements, but falls far short of the two to three orders of magnitude gains required to scale from dozens of tools to tens of thousands.


1.2 Solving the Selection Problem
Who addresses these challenges by providing a dedicated discovery service—essentially a search engine over augments that LLMs can query. The interaction flow is straightforward: a user poses a question through any chat interface, the orchestrating system consults Who, and Who returns relevant augments ranked by suitability.
This architecture resolves each of the problems identified above:
Shared index instead of individual installation. Rather than requiring each user to maintain their own collection of installed augments, Who maintains a shared, continuously updated index—analogous to how web search engines index the web. Augments become discoverable the moment they are indexed, without any action required from end users.


Rich ranking signals beyond descriptions. By relocating selection logic from the LLM to a specialized search service, Who can leverage decades of information retrieval research. For instance, rather than relying solely on augment descriptions, the index can incorporate representative queries where each augment excels or underperforms. Centralizing selection requests also enables continuous improvement based on aggregate usage patterns.


Scalability unconstrained by context windows. Once selection moves outside the LLM, context window limitations no longer apply. Search engines scale sublinearly and routinely operate over tens of billions of documents. Combined with hybrid scoring techniques that reserve LLM-based evaluation for final ranking, this achieves high precision without incurring the full cost of LLM inference. The key insight is that augment selection is fundamentally a ranking problem—one that does not require the full generality of language models.



2. Protocol Overview
Who exposes a single primary operation: who.
Given a natural language query, who returns a ranked list of augments capable of handling the request. Each result includes sufficient information for the caller to invoke the augment directly.
The protocol is transport-neutral. While this specification uses HTTP for its primary examples, the core JSON structures remain identical whether transmitted via HTTP, MCP, WebSockets, or other transports.

A request to Who consists of two sections: query (required) and meta (optional).

4.2 Results Array
For successful responses, the results array contains augment objects ranked by relevance to the query.
Each augment object includes protocol, endpoint, score, and definition fields. The definition contains protocol-specific metadata (A2A Agent Cards, MCP tool definitions, etc.) as described in Section 5.
4.3 Referrals Array
Responses MAY include a referrals array containing pointers to other Who servers that might have additional relevant results. See Section 11 for details on federation and referrals.

9. Integration Patterns
This section illustrates common patterns for integrating Who into orchestration systems.
9.1 Basic Discovery Flow
A typical single-augment interaction proceeds as follows:
User asks: "Find me a good Italian restaurant nearby"
Orchestrator calls Who with the query
Who returns ranked augments (e.g., OpenTable, Yelp, Google Maps)
Orchestrator selects the top-ranked result
Orchestrator invokes the selected augment using the provided endpoint and protocol
Response is returned to the user
9.2 Multi-Augment Composition
For queries requiring multiple capabilities, the orchestrator may issue multiple discovery requests:
User asks: "Plan a dinner date—find a restaurant and book a car"
Orchestrator calls Who for restaurant reservation capabilities
Orchestrator calls Who for car service capabilities
Orchestrator invokes both augments and composes the results
9.3 Fallback Handling
When the top-ranked augment fails or returns unsatisfactory results, the orchestrator can fall back to lower-ranked alternatives:
Orchestrator invokes the top-ranked augment
Augment returns a failure or inadequate results
Orchestrator invokes the second-ranked augment
Process continues until a satisfactory result is obtained or all candidates are exhausted

10. Security Considerations
10.1 Augment Verification
Callers SHOULD verify augment endpoints before invocation. Who indexes augments but does not guarantee their security, authenticity, or availability. Callers are responsible for establishing appropriate trust before invoking returned augments.
10.2 Input Validation
Callers MUST validate that their requests conform to any inputSchema specified in custom invocation details. Malformed requests may produce unexpected behavior or security vulnerabilities.
10.3 Rate Limiting
Who implementations SHOULD enforce rate limiting to prevent abuse and ensure fair access across clients.

11. Federation
Who servers can be organized hierarchically, analogous to DNS. This enables local resolution, policy enforcement, and trust differentiation across organizational boundaries.
11.1 Hierarchical Resolution
A typical deployment might include:
Local Who server — An enterprise-operated instance that indexes internal agents (expense tools, knowledge bases, proprietary services). This is the first resolver consulted.


Upstream Who server — A broader service indexing publicly available agents. The local instance delegates to this when it cannot satisfy a query internally.


Root Who servers — Analogous to DNS root servers, these are well-known public Who servers that provide comprehensive or curated indexes. Multiple root Who servers may exist, differentiated by coverage, trust policies, or domain specialization.


11.2 Trust and Policy
Hierarchical deployment enables several trust and policy patterns:
Curation — An enterprise Who server can restrict results to agents that meet compliance requirements (HIPAA, SOC2, GDPR).
Prioritization — Internal agents can be ranked higher than external alternatives for equivalent capabilities.
Blocking — Certain agents or categories can be excluded based on organizational policy.
Provenance — Results can indicate which Who server vouched for them, enabling callers to make trust decisions.

11.5 Resolution Behavior
Who implementations have discretion in how they handle upstream delegation:
Transparent delegation — The local Who server queries upstream services automatically and merges results before responding. Callers see a unified result set with source fields indicating provenance.


Referral-based delegation — The local Who server returns its own results plus referrals, leaving the caller to query upstream services directly.


Hybrid — The local Who server may transparently delegate to trusted upstream services while returning referrals for others.


The choice depends on factors such as latency requirements, trust relationships, and policy constraints.

12. Discovery
This section addresses how augments are discovered and entered into Who indexes. Just as the web has distinct mechanisms for discovery—DNS for hosts, sitemaps for pages, robots.txt for crawler directives—the agent ecosystem requires analogous mechanisms for advertising capabilities.
We deliberately decouple discovery from the /who protocol. The /who operation queries an index; discovery populates that index. Different deployment contexts will use different discovery mechanisms:
Public web: Sites advertise their agentic capabilities via well-known files and sitemap-like indexes, which Who servers can crawl.
Enterprise: Organizations may use internal registries, service meshes, or configuration management systems to register agents.
Federated: Upstream Who servers may aggregate and republish indexes from downstream sources.
This section specifies the discovery mechanism for the public web. Enterprise and federated discovery are implementation-specific and outside the scope of this specification.
12.1 Design Principles
The discovery mechanism follows these principles:
Alignment with existing standards: We build on established patterns (robots.txt, sitemaps, .well-known URIs) and align with emerging agent protocols (A2A Agent Cards, MCP) where possible.


Declarative, not prescriptive: Sites declare what capabilities they offer; Who servers decide what to index.


Incremental adoption: Sites can advertise a single agent with minimal effort, or publish comprehensive indexes with rich metadata.


Protocol diversity: The mechanism supports multiple augment types (MCP servers, A2A agents, OpenAPI services, Agent Skills) through a unified discovery format.


12.2 Terminology
We introduce the term Capability Manifest as the general name for a machine-readable document that describes one or more capabilities offered by a site. A Capability Manifest can describe:
MCP servers and tools
A2A agents
OpenAPI services
Agent Skills
Other callable augments
The term is deliberately broad, encompassing what A2A calls "Agent Cards" and what MCP discussions have termed "server manifests." A Capability Manifest is to augments what a sitemap is to pages—a declaration of what exists and how to access it.
12.3 Discovery Mechanisms
Who servers can discover Capability Manifests through three complementary mechanisms:
1. Well-Known URI
Sites MAY publish a Capability Manifest at:
https://{domain}/.well-known/capability-manifest.json

This provides a predictable location for single-agent sites or as a canonical entry point.
2. robots.txt Directive
Sites MAY advertise Capability Manifests via a directive in robots.txt:
User-agent: *
Disallow: /private/

Sitemap: https://example.com/sitemap.xml
Agentmap: https://example.com/agents/manifest.json

The Agentmap directive follows the same conventions as Sitemap:
Case-insensitive directive name
Absolute URL value
Multiple directives permitted
This approach supports sites with multiple Capability Manifests or non-standard locations.
3. HTML Meta Tag
Sites MAY include a link to their Capability Manifest in HTML pages:
<link rel="capability-manifest" href="https://example.com/agents/manifest.json">

This enables discovery during web crawling and supports sites where robots.txt modification is not feasible.
12.4 Capability Manifest Format
The Capability Manifest is a JSON document describing one or more capabilities offered by a site. The format cleanly separates the protocol type from protocol-specific metadata, allowing native embedding of A2A Agent Cards, MCP tool definitions, Agent Skills, and other protocol-specific structures.

12.4.4 Protocol-Specific Definitions
Protocol: a2a
For A2A agents, the definition is a native A2A Agent Card:
{
  "protocol": "a2a",
  "endpoint": "https://example.com/.well-known/agent.json",
  "definition": {
    "name": "Travel Concierge",
    "description": "AI-powered travel planning assistant",
    "url": "https://example.com/a2a",
    "version": "1.0.0",
    "capabilities": {
      "streaming": true,
      "pushNotifications": true
    },
    "defaultInputModes": ["text/plain"],
    "defaultOutputModes": ["text/plain", "application/json"],
    "skills": [
      {
        "id": "plan_trip",
        "name": "Trip Planning",
        "description": "Create comprehensive travel itineraries",
        "tags": ["planning", "itinerary"],
        "examples": ["plan a 10-day trip to Japan"]
      }
    ]
  }
}

The definition object is a valid A2A Agent Card as specified by the A2A protocol.
Protocol: mcp
For MCP servers, the definition contains server metadata and the tools array (as returned by tools/list):
{
  "protocol": "mcp",
  "endpoint": "https://api.example.com/mcp",
  "definition": {
    "name": "Weather Service",
    "description": "Weather information and forecasts",
    "version": "1.0.0",
    "tools": [
      {
        "name": "get_current_weather",
        "description": "Get current weather conditions for a location",
        "inputSchema": {
          "type": "object",
          "properties": {
            "location": {
              "type": "string",
              "description": "City name or coordinates"
            },
            "units": {
              "type": "string",
              "enum": ["celsius", "fahrenheit"],
              "default": "celsius"
            }
          },
          "required": ["location"]
        }
      },
      {
        "name": "get_forecast",
        "description": "Get weather forecast for upcoming days",
        "inputSchema": {
          "type": "object",
          "properties": {
            "location": { "type": "string" },
            "days": { "type": "integer", "minimum": 1, "maximum": 14 }
          },
          "required": ["location"]
        }
      }
    ]
  }
}

The definition.tools array follows the MCP tool definition format exactly.
Protocol: openapi
For OpenAPI services, the definition references the specification:
{
  "protocol": "openapi",
  "endpoint": "https://api.example.com",
  "definition": {
    "specUrl": "https://api.example.com/openapi.json",
    "name": "Inventory API",
    "description": "Product inventory management"
  }
}

Protocol: http
For custom HTTP endpoints that don't follow a standard protocol:
{
  "protocol": "http",
  "endpoint": "https://legacy.example.com/api/query",
  "definition": {
    "name": "Legacy Inventory",
    "description": "Query legacy inventory system",
    "method": "POST",
    "contentType": "application/json",
    "inputSchema": {
      "type": "object",
      "properties": {
        "sku": { "type": "string" },
        "warehouse": { "type": "string" }
      },
      "required": ["sku"]
    },
    "authentication": {
      "type": "bearer",
      "description": "Requires API token"
    }
  }
}


Protocol: skill
For Agent Skills (per agentskills.io), the definition contains the YAML frontmatter from the skill's SKILL.md file, converted to JSON:
{
  "protocol": "skill",
  "endpoint": "https://skills.example.com/pptx-creator",
  "definition": {
    "name": "pptx-creator",
    "description": "Create professional PowerPoint presentations following brand guidelines. Use when asked to make slides, presentations, or pitch decks.",
    "license": "MIT",
    "compatibility": "python3, python-pptx",
    "allowed-tools": "Bash(python:*) Read Write",
    "metadata": {
      "author": "example-org",
      "version": "1.2.0",
      "tags": ["presentations", "documents", "office"]
    }
  }
}

The endpoint for skills points to the skill package location—a folder containing:
SKILL.md — Required: instructions and metadata
scripts/ — Optional: executable code
references/ — Optional: documentation
assets/ — Optional: templates and resources

12.5 Indexing Considerations
Who servers processing Capability Manifests SHOULD:
Validate the manifest against the schema before indexing.


Extract skills/tools for search — Index individual skills (A2A) or tools (MCP) with their descriptions and examples to enable fine-grained matching.


Respect robots.txt — Honor any crawl restrictions specified in robots.txt.


Track freshness — Use the version field and HTTP caching headers to detect changes.


Verify endpoints — Optionally probe endpoints to verify availability before indexing.


Who servers MAY augment manifest data with additional signals (query logs, success rates, user feedback) to improve ranking.

B.3 Federated Resolution with Referrals
An enterprise Who server returns internal results and referrals to upstream services.
Request:
{
  "query": {

    "max_results": 5
  }
}

Response:
{
  "_meta": {
    "response_type": "answer",
    "version": "0.1",
    "result_count": 1
  },
  "results": [
    {
      "protocol": "a2a",
      "endpoint": "https://travel.internal.example.com/.well-known/agent.json",
      "score": 95,
      "definition": {
        "name": "Corporate Travel Portal",
        "description": "Book flights using corporate negotiated rates and travel policy",
        "url": "https://travel.internal.example.com/a2a",
        "skills": [
          {
            "id": "book_flight",
            "name": "Flight Booking",
            "description": "Book flights with corporate rates and policy compliance"
          }
        ]
      },
      "source": "https://who.internal.example.com"
    }
  ],
  "referrals": [
    {
      "endpoint": "https://who.nlweb.ai/who",
      "name": "Public Who Server",
      "description": "Comprehensive index of public travel and booking agents",

    },
    {

      "confidence": 92
    }

}

In this example, the enterprise Who server returns the company's internal travel portal (which enforces corporate travel policy) while also providing referrals to public Who servers that index consumer travel services. The caller can decide whether to follow the referrals based on their needs.
